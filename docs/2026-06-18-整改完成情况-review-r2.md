## 1. 阻断项核验：P1-3 并发 lost-update —— ✅ 修复正确

### 1.1 修复实现

`trusted_device_store.go:220-225` 结构体新增 `commitMu sync.Mutex`；`commit`（:254-256）入口 `s.commitMu.Lock(); defer s.commitMu.Unlock()` 串行化整个 read-modify-write 序列。`memMu` 仍只管单次 `mem` 指针 swap 的原子性，只读查找走 `snapshotMem` 的 `memMu RLock` 不阻塞——锁分工正确，不会把只读路径也拖进串行化。

注释（:248-253）写得非常到位，明确区分了"memMu 保证单次 swap 原子性"与"commitMu 保证 RMW 序列隔离性"，并点出"经典原子性≠隔离性"。这是真正理解了问题根源，不是照抄修复指令。

### 1.2 并发测试 —— ✅ 真实有效（经独立反向验证）

开发者声称"去掉 commitMu，16 个设备只剩 1；恢复后 16 个全保留"。我**独立复现**了这个反向验证，过程如下：

1. 临时移除 `commit` 入口的 `commitMu.Lock/Unlock`（用 Edit 改源码）
2. 跑 `TestFileDeviceStoreConcurrentCommitsNoLostUpdate -race -count=5`
3. 结果：**5 次全部失败，`内存设备数 = 1, want 16`** —— lost-update 稳定复现
4. 从备份恢复 `commitMu`，重跑 `-race -count=5` —— **5 次全部通过**

这个验证证明了三件事：
- lost-update 缺陷**真实存在**（不是我误判）
- 并发测试**真的能捕获缺陷**（不是凑数测试，去掉修复就红）
- 开发者声称的"16 个只剩 1"**完全属实**

### 1.3 -race 全量 —— ✅ 通过

修复后 `go test ./go-bridge/ ./core/ -race -count=5 -run 'DeviceStore|ConcurrentCommits|ReadFile|Subdirectory'` 全过，无 DATA RACE 报告。

## 2. 非阻断项核验

### 2.1 P0-1 子目录误拒 —— ✅ 修复语义正确

`handlers.go:3276` 改为 `if requested != root && !pathIsWithinRoot(root, requested) { reject }`。

我核验语义：**授权根始终返回 `root`（workspace 根），没有放宽**。修复只是放宽了"调用方传的 directory 参数"的接受范围——只要 `requested` 等于 root 或在 root 内就接受，越界仍拒绝。最终读取的 path 的越界校验仍由 `resolveAuthorizedReadFilePath`（:3298 起，含 EvalSymlinks + pathIsWithinRoot + TOCTOU）兜底。

所以这个修复没有削弱 P0-1 的安全边界，只是修了"前端浏览子目录时传 directory 会被误拒"的兼容性问题。注释（:3273-3275）也解释了"授权根始终是 workspace 根"。设计正确。

### 2.2 pagination 测试 —— ✅ 已记入剩余风险

第一轮我指出 go-bridge 的 9 个 `TestPaginatedMessages_*`/`TestListSessionsPagination*` 失败是 codex CLI 环境依赖，完成报告当时只提了 opencode 那条。本轮确认：这是开发者本机（有 codex CLI）与我环境（无）的差异，非代码缺陷。这点已记入完成报告剩余风险，建议 CI 固定 codex CLI 版本或标注环境前置——处理得当。
