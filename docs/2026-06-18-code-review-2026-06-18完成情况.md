## 4.4 Review 修复（整改完成情况 review 发现）

P1-3 并发 lost-update（阻断性，已修复）：review 指出 `FileDeviceStore.commit` 注释声称 `commitMu/memMu` 保证序列化，但结构体只有 `memMu`，整个 read-modify-write 序列（snapshotMem→clone→fn→write→swap）无外层串行化。两个并发提交（如同时 approve 设备 A 和 B）都基于同一快照克隆，后 swap 者覆盖前者，先提交的设备记录丢失——经典"原子性≠隔离性"混淆。

修复：`FileDeviceStore` 新增 `commitMu sync.Mutex`，`commit` 入口 `Lock/defer Unlock` 串行化整个 RMW 序列；`memMu` 仍只保证单次指针 swap 原子性、不阻塞只读查找。新增并发测试 `TestFileDeviceStoreConcurrentCommitsNoLostUpdate`（16 并发 AddDevice），并在 `-race` 下验证：去掉 commitMu 测试立即失败（16 设备只剩 1），恢复后 16 个全部保留、内存与磁盘一致。安全相关测试 `-race` 全过；完整 go-bridge 套件在缺 codex CLI 的环境下有 9 个 pagination 测试因环境依赖失败（见剩余风险 §6），非本轮改动。

P0-1 子目录误拒（非阻断，已修复）：review 指出 `matchAuthorizedReadFileRoot` 的 `requested != root` 严格相等会误拒前端浏览 workspace 子目录时传入的合法 `directory`。修复为"requested 等于 root 或在 root 之内即接受，越界才拒绝"；返回的授权根始终是 workspace 根，最终 path 越界校验仍由 `resolveAuthorizedReadFilePath` 完成。新增 `TestReadFileAcceptsSubdirectoryWithinWorkspace`（root/子目录接受、workspace 外拒绝）。
