## 2. 需返工的问题

### 2.1 ⛔ P1-3 设备存储并发 lost-update —— 必须修复

这是本轮发现的**唯一阻断性问题**，且就在 P1-3 自身范围内（讽刺的是，P1-3 本身就是为修并发/崩溃一致性问题而立的门禁）。

**缺陷描述**：

`FileDeviceStore.commit`（`trusted_device_store.go:248-271`）的序列是：

```
snapshotMem()           // 读 mem 指针（memMu RLock 瞬间）
  → current.mu.RLock
  → cloneSnapshot()     // 深拷贝
  → current.mu.RUnlock
  → fn(clone)           // 修改副本（锁外！）
  → AtomicWriteFile     // 写盘（锁外！）
  → memMu.Lock; mem=clone; memMu.Unlock  // swap
```

问题：**整个 read-modify-write 序列没有被任何"提交锁"序列化**。

- `FileDeviceStore` 结构体（:220-224）只有 `memMu sync.RWMutex`，**没有 `commitMu`**。
- 但 commit 的注释（:245-246）写着"commitMu/memMu 保证"——**注释引用了一个不存在的字段**。
- 调用链：`ManagementServer.ServeHTTP`（management_api.go:167）**不加锁**直接分发到 `handleRevokeDevice`(:360)/`handlePairingApprove`(:504)，这些 handler 调用 `DeviceStore.RevokeDevice/ReplaceDevice` 也**不加锁**。`serverMu`(:57) 只保护 Start/Shutdown 等生命周期方法，不覆盖并发写 store。

**后果（lost update）**：两个并发请求（如同时 approve 设备 A 和设备 B，或 approve + revoke）：

```
T1: snapshotMem → clone(v1) → fn:add A → write(A) → mem=clone(A)
T2:              snapshotMem → clone(v1) → fn:add B → write(B) → mem=clone(B)
```

T1 和 T2 都基于 v1 快照克隆，最后 swap 时 **T2 覆盖 T1，设备 A 的变更丢失**。磁盘上只剩 B，内存也只反映 B。A 的配对"成功返回了"但实际没生效。

**为什么 memMu 没救**：`memMu` 只保证 `mem=clone` 这个指针赋值是原子的（不会读到半撕裂数据），但**不保证 read-modify-write 的隔离性**。这是经典的"原子性 ≠ 隔离性"混淆。两个事务各自完整执行了，但交叉重叠导致丢失中间状态。

**触发难度**：中低。配对 approve 是用户/管理 API 触发，理论上可并发（管理后台同时批处理、或 approve 与自动 revoke 竞争）。不是高频路径，但**安全相关**——丢失一条设备记录可能导致"用户以为配对成功但 token 未持久化，重启后认证失败"。

**修复方案**（改动小，约 5-10 行）：

在 `FileDeviceStore` 加一把 `commitMu sync.Mutex`，在 `commit` 入口获取它，覆盖整个 read-modify-write 序列：

```go
type FileDeviceStore struct {
    path     string
    commitMu sync.Mutex   // ← 新增：序列化整个提交序列
    memMu    sync.RWMutex
    mem      *MemoryDeviceStore
}

func (s *FileDeviceStore) commit(fn func(clone *MemoryDeviceStore) error) error {
    s.commitMu.Lock()           // ← 新增：提交全程互斥
    defer s.commitMu.Unlock()
    current := s.snapshotMem()
    // ... 其余不变 ...
}
```

这样 read-modify-write 串行化，T2 必须等 T1 的 swap 完成后才能 snapshotMem（拿到的是含 A 的新快照）。只读查找（`LookupBy*`）走 `snapshotMem` 的 `memMu RLock`，不阻塞提交读，吞吐影响可忽略。

同时**修正注释**（:245-246）让它与实现对齐（或直接按上面补上 commitMu）。

**为什么完成报告和测试没发现**：现有 `trusted_device_store_crash_test.go` 测的是**单线程崩溃一致性**（写盘失败时内存回退），这个场景下实现是正确的——单 commit 内"失败不 swap"确实成立。缺陷只在**并发提交**下暴露，而现有测试没有并发提交用例。建议补一个 `TestFileDeviceStore_ConcurrentCommitsNoLostUpdate`（并发 N 个 AddDevice，断言全部可见）。

**定性**：这是 P1-3 门禁内的回归（门禁要求"事务性提交"，并发丢失更新不是事务性的）。我将其定为 **返工阻断项**，但修复成本极低（加一把锁 + 补一个并发测试 + 改注释）。
